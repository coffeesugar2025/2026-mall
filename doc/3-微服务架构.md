# 微服务架构



![](./img/微服务架构图.drawio.png)







## 一、基础概念层（是什么、优缺点、适用场景）
### 1. 架构演进对比
单体架构 → 垂直拆分 → SOA面向服务架构 → **微服务架构**
1. 单体：所有模块打包一个war/jar，部署简单，迭代、扩容、故障耦合严重
2. SOA：ESB企业服务总线做中转，太重、中心化瓶颈；
3. 微服务：轻量化去中心化、独立自治、粒度更细、每个服务单独部署扩缩。

### 2. 微服务核心定义
- 按**业务域**拆分（用户、订单、商品、支付、库存），一个业务域对应一个独立服务；
- 每个服务：独立代码仓库、独立数据库、独立部署、独立技术栈；
- 服务之间通过网络API通信，互不干扰。

### 3. 优点
- 独立迭代、单独发布，不牵一发而动全身；
- 精准扩容：流量大的订单服务单独加机器，不用整体扩容；
- 技术栈异构：网关用Go、后端Java、数据分析Python灵活选型；
- 故障隔离：单个服务宕机不拖垮整个系统。

### 4. 缺点（落地最大痛点）
- 分布式复杂度陡增：网络超时、重试、事务一致性、调用追踪；
- 运维成本暴涨：服务数量多、部署、监控、日志、流水线复杂；
- 本地调试困难、测试链路变长；
- 额外中间件依赖多，学习成本高。

### 5. 适用/不适用场景
✅ 适合：中大型业务、团队多、迭代频繁、流量波动大、多端接入
❌ 不适合：小项目、初创极简系统、业务稳定几乎不迭代（单体更省事）

## 二、拆分设计方法论（核心设计能力）
### 1. 拆分原则（DDD领域驱动设计是主流标准）
1. 按**限界上下文**拆分，而非技术层（不能Controller一层、Service一层拆分）；
2. 高内聚、低耦合：一个服务只负责自己业务；
3. 单一数据源：微服务尽量独享数据库，禁止多服务共用库表；
4. 粒度适中：不能拆太细（过度微服务，调用爆炸），也不能粗粒度变回单体。

### 2. 常见标准服务拆分示例
- 用户服务：注册、登录、权限、会员、地址
- 商品服务：SPU/SKU、类目、库存、上下架
- 订单服务：创建订单、订单状态、售后、退款
- 支付服务：对接支付宝/微信、流水、回调处理
- 购物车服务、优惠券服务、物流服务、消息通知服务

### 3. 两种拆分误区
1. 技术分层拆分（所有服务共用一张库表）→ 伪微服务；
2. 无限拆分（每个接口一个服务）→ 调用链过长，维护灾难。

## 三、微服务架构完整技术组件栈（整套落地必备）
### （1）API网关层（统一入口）
作用：路由分发、鉴权、限流、熔断、SSL、灰度发布、聚合接口
主流组件：
- Spring Cloud Gateway（Java）、Nginx、Kong、APISIX、Traefik
核心能力：
请求路由、JWT登录校验、IP黑名单、流量控制、跨域处理

### （2）服务注册与发现（解决“去哪调用服务”）
问题：服务多、IP端口动态变化，调用方不能硬编码地址
组件：
- Nacos、Eureka、Consul、etcd、ZooKeeper
核心机制：
服务启动上报地址到注册中心；调用方拉取服务实例列表，负载均衡调用

### （3）负载均衡LB
1. 客户端LB：Spring Cloud LoadBalancer、Ribbon（本地内存维护实例列表）
2. 服务端LB：Nginx、网关层统一负载
策略：轮询、随机、权重、最小连接、IP哈希

### （4）远程服务调用（RPC通信）
两种主流通信方案：
1. HTTP/REST（通用性强）：OpenFeign，跨语言友好
2. RPC高性能二进制：Dubbo、gRPC、Motan，高并发内部调用首选

### （5）配置中心（统一管理配置）
痛点：几十上百个服务，每个服务yml配置分散，改配置要逐个重启
组件：Nacos、Apollo、Spring Cloud Config
能力：
统一配置管理、环境隔离（dev/test/prod）、配置热更新、加密敏感配置

### （6）熔断、降级、限流（高可用防护，防雪崩）
分布式经典问题：级联故障（A调用B，B宕机导致A线程耗尽）
组件：Sentinel、Resilience4j、Hystrix（老旧）
- 熔断：连续失败直接断开调用，快速失败
- 降级：接口不可用时返回兜底默认数据
- 限流：限制QPS，防止突发流量打垮服务
- 仓壁模式：不同调用隔离线程池，互不抢占资源

### （7）分布式事务（跨服务数据一致性）
微服务无法用本地事务，订单、库存、支付跨库更新，数据一致性难题
主流方案：
1. AT（Seata）：无侵入，最常用；
2. TCC：侵入代码，强一致性；
3. SAGA：长事务、流程编排；
4. 本地消息表、可靠消息最终一致性（RabbitMQ/RocketMQ）；
5. 最大努力通知（短信、通知类场景）。

### （8）分布式异步通信（消息队列MQ）
解耦、削峰、异步处理
场景：订单创建后异步发通知、扣积分、日志存储
中间件：RocketMQ、RabbitMQ、Kafka、Pulsar
使用模式：生产者-消费者、广播、死信队列、延迟消息

### （9）分布式链路追踪（排查调用链路）
问题：一次请求跨5~10个服务，报错不知道卡在哪个节点、耗时在哪
组件：SkyWalking、Pinpoint、Jaeger、Zipkin
核心指标：TraceId（整条链路唯一ID）、SpanId、调用耗时、异常堆栈

### （10）分布式日志收集
多服务日志分散在不同机器，无法逐个登录排查
架构：Filebeat → Logstash → Elasticsearch + Kibana（ELK）
轻量方案：Loki + Grafana

### （11）监控告警体系
1. 服务器监控：CPU、内存、磁盘、网络（Prometheus + Grafana）
2. 应用监控：JVM、QPS、错误率、响应耗时
3. 告警：短信、钉钉、企业微信告警，异常主动推送

## 四、数据层设计知识
1. 数据库拆分：每个微服务独立库，禁止跨服务JOIN联表查询；
2. 多数据源场景：分库分表 Sharding-JDBC；
3. 缓存设计：分布式Redis，解决本地缓存数据不一致；
   缓存穿透、击穿、雪崩解决方案；
4. 分布式Session：统一Redis共享登录会话；
5. 分布式ID：雪花算法、Redis自增、发号器（避免多库主键冲突）。

## 五、部署与DevOps运维体系
### 1. 容器化打包
每个微服务打包Docker镜像，环境一致，一处打包到处运行

### 2. 编排调度
K8s(Kubernetes)：自动部署、弹性扩缩、自愈、滚动更新、灰度发布

### 3. CI/CD流水线
GitLab/Gitee + Jenkins / GitLab CI / ArgoCD
代码提交→自动编译→单元测试→构建镜像→自动部署到测试/生产环境

### 4. 环境划分
开发环境dev、测试test、预发staging、生产prod，配置隔离

## 六、微服务高级落地方案
### 1. 服务网格 Service Mesh（Istio）
把限流、熔断、重试、加密、流量治理从业务代码剥离，Sidecar代理接管网络，业务代码无侵入，适合超大规模集群。

### 2. 灰度发布/金丝雀发布
新版本只切10%流量验证，无问题再全量切换，回滚成本极低；
蓝绿部署、滚动发布。

### 3. BFF后端聚合层
前端多端（APP、小程序、H5）需要聚合多个微服务数据，单独搭建BFF层做接口组装，减少前端多次请求。

### 4. 云原生配套
Serverless、容器镜像仓库Harbor、对象存储、云主机弹性伸缩。

## 七、两大主流技术栈体系
### 1. Spring Cloud 生态（Java后端主流）
Gateway + Nacos + OpenFeign + Sentinel + Seata + SkyWalking + Prometheus
### 2. 云原生跨语言体系（Go/Python/PHP）
APISIX/Istio + gRPC + Consul + 消息队列 + K8s，不绑定语言
