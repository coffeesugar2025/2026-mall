# 技术选型

这篇文档回答一个简单但重要的问题:**每个中间件为什么选它,而不是别的?** 对于学习型项目来说,"讲清楚为什么"比"用了什么"更重要——这也正是面试官想听的。

格式约定:每项技术三段式——**场景 · 我们为什么选它 · 备选与代价**。

---

## 服务治理:Nacos

**场景**:需要一个同时提供"服务注册发现"和"动态配置中心"的组件。

**为什么选它**:

- 一套组件同时搞定两件事,减少运维负担(Eureka 只做注册,Config 要单独部署)
- 配置变更支持实时推送,`@RefreshScope` 不需要重启
- 中文文档完善,国内生态更成熟,和 SCA(Spring Cloud Alibaba)无缝集成
- UI 控制台直观,新手上手 30 分钟

**备选与代价**:

| 备选 | 问题 |
|-----|------|
| Eureka | 停止维护、无配置中心、集群脑裂问题 |
| Consul | 部署运维复杂,国内网络不稳定 |
| ZooKeeper | CP 模型,注册中心更需要 AP,且 API 太低级 |
| Apollo | 只做配置,还要再搭一个注册中心 |

---

## 网关:Spring Cloud Gateway (WebFlux)

**场景**:统一入口,做路由、鉴权、限流、跨域。

**为什么选它**:

- Spring 官方主推,和 Spring Cloud 生态天然整合
- 基于 Netty + Reactor,非阻塞 I/O,单节点吞吐远高于 Zuul 1.x
- Filter 模型清晰(WebFilter + GlobalFilter),扩展成本低
- 支持 WebSocket 路由

**备选与代价**:

| 备选 | 问题 |
|-----|------|
| Zuul 1.x | 阻塞 I/O,吞吐低,已被 Gateway 取代 |
| Kong / APISIX | 功能强大但是独立进程,和 Java 生态割裂,学习曲线更陡 |
| Nginx + Lua | 性能最强但不便做业务鉴权,改动成本高 |

本项目网关里最值得学的一点是:**用响应式 WebFilter 写同步 Redis 调用时如何不阻塞事件循环**。参见 [专题 02](../07-亮点技术专题/02-网关鉴权实战.md)。

## 分布式事务:Seata TCC

**场景**:热门车次下单时,**跨 `rs-order` 和 `rs-ticket` 两个服务**,既要预占座位又要扣减余票,同时还要落库订单。一旦某一步失败,全部回滚。

**为什么选 TCC 而不是 AT**:

| 对比项 | AT 模式 | **TCC 模式** |
|-------|---------|------------|
| 侵入性 | 低(自动生成反向 SQL) | 高(需手写 Try/Confirm/Cancel) |
| 性能 | 受全局锁影响 | **无全局锁,吞吐更高** |
| 适用场景 | CRUD 为主,低并发 | **秒杀、高并发、复杂业务** |
| 脏读风险 | 默认不隔离,需加 `@GlobalLock` | **完全由业务控制** |

秒杀场景下,AT 模式的全局行锁会让热门车次变成"**串行下单**",TCC 通过 Redis 预扣 + 业务补偿,把一致性成本前置到 Try 阶段,主流程始终无锁。

**备选与代价**:

- **最大努力通知 / 本地消息表**——实现简单但一致性更弱,出事靠人工补偿
- **Saga**——长事务场景好用,但步骤多时编排复杂,本项目步骤少不划算
- **XA**——强一致但性能差,MySQL XA 在主从下还有坑

代价:TCC 的 Try/Commit/Rollback 都要手写,且必须保证幂等。本项目通过 `TICKET_ORDER_ID:{branchId}` 和 `TICKET_DEDUCTION_TAG:{branchId}` 这种"分支事务级 Redis key"来实现幂等标记,详见 [专题 01](../07-亮点技术专题/01-秒杀一致性.md)。

## 缓存:Redis + Redisson + Lua

**场景**:

1. Token 存储(JWT 反向索引、管理员 Session)
2. 热点数据缓存(热门车次、商品详情)
3. 分布式锁(下单防重、座位占用)
4. **秒杀场景的原子扣减**

**为什么是这套组合**:

- Redis 本身足够快,单机 10W+ QPS
- Redisson 封装了"看门狗续期"的分布式锁,比手写 `SETNX` 健壮得多
- Lua 脚本让"查余票 + 扣减"在 Redis 侧原子执行,避免"查完准备扣减时被别人抢先"

看项目里的 `CheckRemainingTicket.lua`:

```lua
local key = KEYS[1]
local needCount = tonumber(ARGV[1])
local currentStock = tonumber(redis.call('GET', key))
if currentStock == nil or currentStock < needCount then
    return 0
end
redis.call('DECRBY', key, needCount)
return 1
```

短短 8 行,就是秒杀一致性的第一道防线。

**备选与代价**:

- Memcached 不支持 Lua,也无持久化
- 本地缓存(Caffeine)不适合分布式场景

## 消息队列:RabbitMQ

**场景**:

1. TCC Commit 阶段异步落库订单
2. 订单成功后异步发放积分
3. 订单超时取消的延迟消息
4. 积分兑换成功异步扣积分

**为什么选 RabbitMQ 而不是 Kafka / RocketMQ**:

| 对比项 | **RabbitMQ** | Kafka | RocketMQ |
|-------|-------------|-------|---------|
| 事务消息 | ✅ publisher confirm | ❌ 原生不支持 | ✅ 半消息 |
| 延迟消息 | ✅ 插件支持 | ❌ 需自己实现 | ✅ 原生支持 |
| 吞吐 | 中等(万级) | **极高(百万级)** | 高(十万级) |
| 部署复杂度 | **低** | 中(依赖 ZK) | 中 |
| 适用 | **业务消息** | 日志流、大数据 | 金融级事务 |

对于"业务消息为主、吞吐要求中等、需要延迟消息"的铁路系统,RabbitMQ 是性价比最高的选择。如果项目演进到每日千万级订单,再考虑换 RocketMQ。

---

## 搜索:Elasticsearch 8

**场景**:积分商城商品搜索,支持关键词、分页、过滤。

**为什么选它**:

- 全文检索场景 MySQL 的 LIKE 查询会全表扫描,10w+ 商品就扛不住
- ES 的倒排索引 + IK 中文分词能满足"搜索'毛巾'也能匹配'浴巾'"这种近义词需求
- Spring Data Elasticsearch 封装成熟

**代价**:运维复杂度上升(JVM 内存、索引策略、版本升级)。

本项目为了降低上手门槛,**ES 是可选依赖**——不启用只会让商城搜索退化为数据库精确查询,不影响主流程。

---

## 分布式调度:XXL-Job

**场景**:订单超时自动关单、每日车次库存预热到 Redis。

**为什么选它**:

- UI 完善、中文文档齐全
- 失败告警、日志、重试都有成品
- 单机就能部署,比 Quartz 集群轻

**备选**:Spring 自带 `@Scheduled` 无法做跨节点调度(多实例会重复执行),所以必须用 XXL-Job 或 ElasticJob。

---

## 实时通信:Netty + WebSocket

**场景**:AI 客服的实时对话(用户端 + 人工客服端),需要流式输出模型 Token。

**为什么是 Netty 而不是 Spring WebSocket**:

- 原生 Netty 性能更高,连接数可到 10w+
- 自己控制 Handler Pipeline,方便做**消息路由**(用户 ↔ AI ↔ 人工客服三方转发)
- 更贴近面试八股(NIO、Reactor、ByteBuf),这是本项目的"学习价值"定位决定的

项目里的 `NettyConfig` 启动两个 WS 端点:`/user` 和 `/agent`,分别对应 [`UserWebSocketHandler`](../../RailwaySystem-Backend/rs-service/rs-assistant/src/main/java/com/rs/handler/UserWebSocketHandler.java) 和 [`AgentWebSocketHandler`](../../RailwaySystem-Backend/rs-service/rs-assistant/src/main/java/com/rs/handler/AgentWebSocketHandler.java)。

---

## AI 能力:LangChain4j

**场景**:AI 客服问答,需要会话记忆、工具调用、模型可切换。

**为什么选它**:

- LangChain 的 Java 端对齐版,API 设计熟悉度高
- 原生支持**会话记忆(ChatMemory)**、**工具调用(Tools)**、**流式响应(StreamingChatLanguageModel)**
- 模型适配层丰富(OpenAI、Ollama、Qwen/DashScope、Zhipu、Doubao...),本项目默认对接阿里百炼

**备选**:

- Spring AI——太新,API 还在大改,生态不如 LangChain4j
- 直接调 HTTP——要自己做会话、工具、流式,不划算

## 为什么不用这些(常被问)

| 没用 | 原因 |
|------|------|
| Sentinel | 网关当前只有基础限流,未接熔断降级——列入 Roadmap |
| SkyWalking / Zipkin | 链路追踪未接入——列入 Roadmap |
| 配置中心 Apollo | 已用 Nacos 覆盖,没必要再引入 |
| Dubbo | 项目定位是 Spring Cloud 全家桶学习,统一走 Feign |
| ShardingSphere | 单库已够用,分库分表不是学习重点 |
| 容器化(K8s) | 学习型项目先跑通为主,容器化列入 Roadmap |

## 升级路径建议

如果你想继续扩展这个项目作为进阶练手,推荐顺序:

1. 接入 Sentinel,把网关限流升级到熔断降级
2. 引入 SkyWalking,观察跨服务调用链
3. 做 Docker Compose 一键启动,降低协作门槛
4. 把 `rs-assistant` 拆出独立 Domain,做 Agent 工具编排(调用其它服务真正查订单)
5. 秒杀场景接 JMeter 压测,写一篇《QPS 从 500 → 5000 的优化记录》

这些 Roadmap 已经在 [ROADMAP.md](../ROADMAP.md) 有列。
